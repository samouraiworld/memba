# MEMBA-2026-001 ‚Äî Cross-chain ADR-036 signature replay (AUTH-CHAINID-01)

> **Update (2026-06-12, A2 sign-bytes work):** the auth mechanism below described an **ADR-036**
> sign document. In practice **Adena cannot produce ADR-036** (`MsgSignData` is unsupported ‚Äî it
> signs only tx-shaped documents), so the frontend was sending an empty signature in production and
> the ADR-036 verification path was never exercised. `MakeToken` now verifies a **tx-shaped login
> proof** over gno-canonical sign-bytes (`internal/auth/login_challenge.go`); the chain binding is
> preserved by embedding `chain_id` in the signed sign-doc rather than the ADR-036 template. The
> historical claims below that "clients send `chain_id`" and that a "24h grace closes" reflect the
> *intended* ADR-036 rollout, **not** observed production behavior (logins were unsigned, gated by
> `MEMBA_ALLOW_UNSIGNED_AUTH`). The legacy `MakeADR36SignDoc` helper remains only until phase-2
> removal. See `docs/planning/MEMBA_AAA_A2A3_SIGNBYTES_DESIGN.md` ¬ß5 (A2).

## Summary

The backend constructed the [ADR-036](https://docs.cosmos.network/main/architecture/adr-036-arbitrary-signature) sign document with `chain_id` hard-coded to the empty string. As a consequence:

1. A wallet signature produced for "Login to Memba Multisig Service" on chain `test12` is **bit-identical** to one produced for the same payload on chain `gnoland1` (or any other chain serving the same Memba codebase). The signature carries no chain context.
2. Auth tokens issued by the server were **not chain-scoped** ‚Äî `Token` had no `chain_id` field. Once Memba runs on multiple chains, a token issued for test12 could be presented to a gnoland1 instance and accepted on signature validity alone.

The vulnerability has been latent since the initial auth implementation. It became relevant in v7.1 planning because betanet (`gnoland1`) deploys make multi-chain operation real.

## Impact

* **Today (single-chain deployment, `test12`)**: theoretical ‚Äî there is only one Memba backend (memba-backend.fly.dev) and it talks to `test12` only. An attacker can still capture a user's ADR-036 signature within the 5-minute challenge TTL and replay it, but on a single-instance deployment they can already do that without the chain confusion.
* **Once betanet deploys (Phase 5)**: real. A signature obtained from a user authenticating on test12 would be valid for authenticating that same user against a separate gnoland1-bound backend that shares this code.
* **In general**: violates least-privilege. A user signing "log me into Memba on chain X" should not implicitly authorize "log me into Memba on chain Y."

No evidence of in-the-wild exploitation.

## Fix

`v6.0.2` lands the following changes (see PR0a):

1. **`MakeADR36SignDoc(data, signer, chainID)`** ‚Äî `chain_id` is now passed in and JSON-encoded into the signDoc. `chain_id:""` is no longer the default; an explicit chain identifier is required.
2. **`Challenge.chain_id` and `Token.chain_id` (proto, field 5 each)** ‚Äî server records the chain a challenge/token was issued for; these fields are part of the server-signed payload, so they cannot be tampered with.
3. **`TokenRequestInfo.chain_id` and `GetChallengeRequest.chain_id` (proto, fields 6 and 2)** ‚Äî clients send the chain they intend to authenticate for.
4. **`MakeToken(..., defaultChainID)`** ‚Äî accepts the server's configured chain (env `GNO_CHAIN_ID`) as a fallback for clients that have not yet been updated to send `chain_id`.
5. **`ValidateToken(..., expectedChainID)`** ‚Äî rejects tokens whose recorded `ChainId` does not match the server's expected chain. Empty values on either side are accepted (legacy grace, see below).
6. **`Challenge.ChainId` cross-check during `MakeToken`** ‚Äî if the challenge was minted bound to a chain, the eventual auth must be for that same chain.

## Graceful rollout (24h legacy grace window)

Clients shipped before PR0b do not send `chain_id` in their `TokenRequestInfo`. To avoid logging out every user the moment `v6.0.2` deploys, the server applies a graceful rotation policy:

| Scenario | Behavior |
|----------|----------|
| Client sends `chain_id` matching server's `GNO_CHAIN_ID` | Accepted; modern path. |
| Client sends `chain_id` mismatching server | **Rejected** (`"challenge was issued for a different chain"`). |
| Client sends empty `chain_id`, server has `GNO_CHAIN_ID` set | Server signs the modern signDoc with its configured chain; if the wallet's ADR-036 signature was constructed against `chain_id:""` (legacy client), the server retries verification with the legacy signDoc and logs a deprecation warning. |
| Token previously issued without `ChainId` (pre-`v6.0.2`) | Accepted by `ValidateToken` for the lifetime of the existing 24h token TTL, with a deprecation log. |

The grace window naturally closes after **24 hours** because all pre-fix tokens have expired by then. PR0b updates the frontend to send `chain_id` explicitly, eliminating the deprecation path on the happy flow.

## Regression tests

Added in `backend/internal/auth/crypto_test.go`:

* `TestADR36SignDocEmbedsChainID` ‚Äî canary asserting `signDoc(payload, signer, "test12") != signDoc(payload, signer, "gnoland1")` and `!= signDoc(payload, signer, "")`.
* `TestADR36SignDocChainIDJSONEscape` ‚Äî hostile `chain_id` values cannot break the signDoc structure (JSON-marshalled in).
* `TestTokenChainBinding` ‚Äî issued token carries the chain; cross-chain `ValidateToken` rejects.
* `TestTokenChainBindingLegacyGrace` ‚Äî pre-fix tokens are still accepted during the grace window.
* `FuzzMakeADR36SignDoc` ‚Äî seeded fuzz over arbitrary `chainID`/`signer`/`data` triples; asserts well-formed JSON.

## Coordinated disclosure

Adena wallet shares the ADR-036 auth surface with Memba. The Adena team is informed in parallel with this advisory's publication via `docs/comms/v7.1-adena-disclosure.md` so they can audit their own multichain handling.
