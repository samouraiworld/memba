# Memba v2.x — Session Conventions

> **This file MUST be read at the start of EVERY development session.**
> It defines the mandatory rules for all v2.x development work.

---

## 1. Session Startup Checklist

Every new session must begin with these steps, in order:

```
[ ] 1. Read this file (SESSION_CONVENTIONS.md)
[ ] 2. Read docs/planning/MASTER_ROADMAP.md for current milestone status
[ ] 3. Read the HANDOFF.md of the current milestone (docs/planning/milestones/v2.0-<phase>/)
[ ] 4. git pull origin dev/v2 && git status
[ ] 5. cd frontend && npm test (verify baseline)
[ ] 6. cd frontend && npx tsc --noEmit (verify types)
[ ] 7. cd frontend && npm run build (verify build)
[ ] 8. cd backend && go test -race -count=1 ./... (verify backend)
```

Only proceed with development after ALL 8 checks pass.

---

## 2. Git Workflow (Strict)

### Branch structure
```
main (v1.7.x — production, never commit directly)
  └── dev/v2 (integration branch — holds all v2.0 work)
       ├── feat/v2.0-alpha/plugin-architecture
       ├── feat/v2.0-alpha/deployment-pipeline
       ├── feat/v2.0-alpha/member-proposals
       ├── feat/v2.0-beta/board-realm-template
       ├── feat/v2.0-beta/board-ui
       └── ...
```

### Rules
- ❌ NEVER commit to `main`
- ❌ NEVER commit to `dev/v2` directly
- ✅ Always create a feature branch from `dev/v2`
- ✅ Branch naming: `feat/v2.0-<phase>/<feature-name>`
- ✅ When feature is complete: merge into `dev/v2` (squash merge)
- ✅ When milestone is complete: create PR from `dev/v2` → `main`
- ✅ Read `/git-policy` workflow before any git operations

### Hotfix path (for production v1.7.x bugs)
```
main → hotfix/v1.7.x-<issue> → merge to main → tag v1.7.x
```
Hotfixes go directly to `main`. v2 development is unaffected.

## 3. Quality Gates (Mandatory Per Feature)

### 3a. 11-Perspective Cross-Audit

Every feature must document conclusions from these perspectives:

| # | Perspective | Focus |
|---|-------------|-------|
| 1 | 🔒 CTO | Architecture, scalability, debt |
| 2 | 🛡️ CSO | Security model, data exposure |
| 3 | 🔴 Red Team | Attack vectors, injection, DoS |
| 4 | 🔵 Blue Team | Defence-in-depth, monitoring |
| 5 | 🎯 Black Hat | Exploits, privilege escalation |
| 6 | 🎨 UX/UI | Mobile responsive, accessible, smooth |
| 7 | ⚙️ Gno Core | Crossing syntax, ABCI, gas |
| 8 | 📢 DevRel | Documentation quality, onboarding |
| 9 | 💻 Fullstack | Code quality, DRY, bundle size |
| 10 | 💰 DeFi User | Financial UX, fees, slippage |
| 11 | 🏛️ DAO User | Governance UX, proposal clarity |

### 3b. Test Gate

| Check | Command | Requirement |
|-------|---------|-------------|
| Unit tests | `cd frontend && npm test` | All pass, ≥90% coverage for new code |
| E2E tests | `cd frontend && npx playwright test` | All pass Chromium + Firefox |
| Backend tests | `cd backend && go test -race -count=1 -cover ./...` | All pass, 0 races |
| Lint | `cd frontend && npm run lint` | 0 errors, 0 warnings |
| TypeScript | `cd frontend && npx tsc --noEmit` | 0 errors |
| Build | `cd frontend && npm run build` | Clean, bundle < 500KB |
| Security | `cd backend && govulncheck ./... && gosec ./...` | 0 findings |

### 3c. Documentation Gate

| File | When to update |
|------|---------------|
| `CHANGELOG.md` | Every feature merge |
| `ROADMAP.md` | Milestone status changes |
| `README.md` | User-facing feature changes |
| `docs/ARCHITECTURE.md` | Data flow changes |
| `docs/API.md` | New RPC endpoints |
| Plugin README | Each plugin: `docs/plugins/<name>.md` |
| Milestone HANDOFF.md | End of every session |

## 4. Session End Checklist

Before ending any development session:

```
[ ] 1. All tests pass (unit, E2E, lint, types, build)
[ ] 2. CHANGELOG.md updated (if changes were made)
[ ] 3. Update milestone HANDOFF.md with:
        - What was completed
        - What remains
        - Gotchas / learnings
        - Current test results
[ ] 4. Commit and push to feature branch
[ ] 5. dev/v2 must compile and pass tests (if branch was merged)
```

---

## 5. Milestone Lifecycle

Each milestone follows this lifecycle:

```
1. BRIEF.md exists with scope and acceptance criteria
2. Create feature branch(es)
3. Implement features (tests + docs alongside)
4. 11-perspective audit documented
5. All quality gates pass
6. Merge to dev/v2
7. Write HANDOFF.md
8. Update MASTER_ROADMAP.md status
```

---

## 6. Code Standards

### Frontend
- React functional components with hooks
- TypeScript strict mode (`noEmit` check must pass)
- Plugins: `frontend/src/plugins/<name>/` — lazy loaded, isolated
- CSS: vanilla CSS, no Tailwind
- IDs: unique, descriptive, for E2E targeting

### Backend
- Go standard library preferred
- `go vet`, `gosec`, `govulncheck` clean
- Protobuf for API contracts

### Realm code (Gno)
- Generated by `daoTemplate.ts` / `boardTemplate.ts`
- Must include helper functions for MsgCall compatibility
- Uses `localDAO` (not `DAO`) for function calls
- All code generated must be testable via `gno test`

---

## 7. Stealth Mode

> v2 development is **private and discreet**.

- No public GitHub issues or milestones for v2 work
- `dev/v2` branch is pushed quietly
- Feature branches use descriptive but neutral names
- No social media announcements until v2.0 is merged and deployed
- PR to `main` only when v2.0 is fully complete and tested
