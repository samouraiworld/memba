## 2. What the review process caught along the way (already fixed pre-merge)

The gate was not ceremonial. Highest-value catches, each fixed before its PR merged:

- **W2.2 (critical):** gno encodes "never-transacted address" as an ABCI *error object* (`/std.UnknownAddressError`) — the naive fail-loud would have bricked every fresh multisig's **first** transaction and pinned first-tx/faucet quests forever. Caught by cross-referencing the backend's own live-RPC fixtures.
- **W1.5:** roster-first membership lookup would have let a governance-removed member keep seeded channel roles (incl. `admin`) forever — replaced with a live `parent.IsMember` gate, proven by driving a real on-chain removal in the gno-VM test.
- **W2.1:** deploys must never auto-retry (re-sign popup loop → "package already exists"), and the `changedNetwork` re-validation window itself had to fail closed, not just the end state.
- **W2.3 / W3.4:** runbook steps that were unexecutable in the actual container (no sqlite3 CLI) and recovery pointers to a backup mechanism the same PR retired.
- **W3.3:** one factual error in ~40 sampled doc claims (`GetNFTActivity` limits) — from a stale proto comment rather than the handler.

## 3. Findings from this final pass

**MED — multisig quorum display counts unverified signatures (log-only window).**
`SignTransaction` stores a signature even when `VerifyMultisigMemberSignature`
fails (by design while `MEMBA_ENFORCE_MULTISIG_SIG_VERIFY` is unset — the
tracked A3 posture), and both `CompleteTransaction`'s threshold gate and the
UI's broadcast-enable count **raw rows**. The chain independently verifies the
real multisig signatures, so **no fund movement is possible** — but members can
be shown "quorum reached" that doesn't cryptographically exist (e.g. a bogus
pasted gnokey sig). *Action queued:* per-signature `verified` flag (mirroring
`Transaction.verified`) + "N/M verified · K submitted" display; the gating
change itself stays coupled to the A3 fix. (Follow-up chip filed.)

**LOW (notes, below reporting threshold):**
- `escrowTemplate` `FundMilestone` accepts overpayment with no excess sweep — self-inflicted only (the app always sends exact); defensive `!=` reject is a cheap future hardening.
- `CreateTransaction` trusts client `account_number`/`sequence` — fails safe at broadcast (sequence mismatch), not an authz issue.
- W1.1 nuance: list-shaped inputs (roles/categories/channel names) are silently *filtered* rather than thrown; injection is prevented either way, but an invalid role vanishes instead of surfacing in the wizard.
- Two stale test counts outside W3.3's claimed scope (`GNO_CORE_COMPAT.md`, `DEPLOYMENT_RUNBOOK.md`) — **fixed in this PR**.
